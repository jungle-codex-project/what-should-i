# 팀 컨벤션

이 문서는 팀원이 공통으로 따라야 하는 저장소 작업 규칙을 정리한 문서입니다.

## 커밋 메시지

### 1. 작업 유형 지정

- 작업 유형은 소문자로만 작성한다.

| 작업 유형 | 의미 |
| --- | --- |
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `style` | 코드 formatting, 세미콜론 누락, 코드 자체의 변경이 없는 경우 |
| `refactor` | 코드 리팩토링 |
| `test` | 테스트 코드, 리팩토링 테스트 코드 추가 |
| `chore` | 의존성 수정, 간단한 변경 |
| `db` | 스키마 변경 사항 |
| `hotfix` | 운영 이슈 또는 긴급 장애 수정 |

### 2. 가능한 한 하나의 작업 단위로 커밋

- 하나의 커밋은 하나의 목적만 담는다.
- 관련 없는 변경을 한 커밋에 섞지 않는다.

### 3. 메시지는 한글로 작성

- 형식은 `작업유형: 작업 내용`으로 작성한다.
- 콜론 뒤에는 공백 1칸을 둔다.
- 첫 줄 제목은 가능한 한 짧고 명확하게 작성한다.

```text
feat: 구글 로그인 구현
```

### 4. 변경 사항이 많을 경우 추상화해서 작성하고 설명 추가

- 제목에는 변경 목적을 요약한다.
- 필요한 경우 본문은 한 칸 띄운 뒤 여러 줄로 설명한다.

```text
refactor: 피드 조회 조건 추가

deletedAt이 null인 조건
cursor보다 이전 조건
...
```

### 5. 커밋 메시지 작성 규칙

- 첫 줄만으로도 변경 목적이 이해되게 작성한다.
- 첫 줄은 가능하면 50자 내외로 유지한다.
- 부가 설명이 필요 없으면 본문은 생략한다.
- 본문이 있으면 무엇을 했는지보다 왜 바꿨는지를 우선 설명한다.

## 브랜치 네이밍

### 1. 작업 유형 지정

- 작업 유형은 소문자로만 작성한다.

| 작업 유형 | 의미 |
| --- | --- |
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `style` | 코드 formatting, 세미콜론 누락, 코드 자체의 변경이 없는 경우 |
| `refactor` | 코드 리팩토링 |
| `test` | 테스트 코드, 리팩토링 테스트 코드 추가 |
| `chore` | 의존성 수정, 간단한 변경 |
| `db` | 스키마 변경 사항 |
| `hotfix` | 운영 이슈 또는 긴급 장애 수정 |

### 2. 브랜치 이름 형식

- 브랜치 이름은 `작업유형/작업내용` 형식으로 작성한다.
- 작업 내용은 영문 소문자와 하이픈(`-`)만 사용한다.
- 브랜치 하나에는 하나의 기능 또는 하나의 목적만 담는다.

```text
feat/implement-google-login
hotfix/fix-login-redirect
```

### 3. main 브랜치 운영 규칙

- feature 단위로 브랜치를 만들어 작업한다.
- `main` 브랜치에서는 직접 작업하지 않는다.
- `main` 브랜치에는 pull request로만 반영한다.
